# CMPD — Commit · Merge · Push · Deploy

Full release workflow for the upsquad website. Invoked when the user says "CMPD" or `/cmpd`.

## Preconditions
- User has authorized the release.
- SSH to `root@72.61.245.97` is working.
- You are on a feature branch (the `block-edits-on-main.sh` hook blocks edits on `main` — create one first if needed).
- All intended code changes are in the working tree.

## 1. Commit

- Run `git status -s` and `git diff` to understand the changes.
- Scan for anything that looks like a secret (`.env`, `credentials.json`, `token.json`). Refuse to stage those.
- Stage files **by name** (`git add path/to/file`) — never `-A` or `.`.
- Draft a concise subject line focused on the *why*, optional body for context.
- Append the standard trailer: `Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>`.
- Commit via a HEREDOC so the message format is preserved.

## 2. Merge

- `git checkout main`
- `git merge --ff-only <feature-branch>` — refuse to create merge commits. If fast-forward fails, stop and report.

## 3. Push

- `git push origin main`. Verify the before/after SHAs from the output.

## 4. Deploy (to the VPS)

SSH in and run these steps. **Do not use `tools/deploy.sh`** — that script targets a future Docker-based deploy at `/opt/upsquad` which isn't live yet.

```bash
ssh root@72.61.245.97
cd /root/upsquad

# The VPS working tree occasionally has stray untracked artifacts
# (Next.js next-env.d.ts, client/tsconfig.json, uncommitted lockfile nudges).
# Stash them so the pull is clean.
git stash push -m 'pre-deploy-stash' || true

git pull origin main
```

Then decide what to rebuild based on what changed:

| Changed | Action |
|---|---|
| anything in `client/` | `cd /root/upsquad/client && NODE_ENV=production npm run build` — Next.js has `output: 'export'` + `distDir: '../server/public'`, so the build writes the static site directly into `server/public/` |
| `server/index.js`, `server/routes/`, `server/lib/`, `server/views/`, or `server/package.json` | `cd /root/upsquad/server && npm install && pm2 restart upsquad` |
| only docs / workflows / tooling | nothing — no rebuild needed |

Static-only changes do **not** need a pm2 restart (Express re-reads `server/public/` on every request).

## 5. Verify

Hit the live URL for the specific thing that changed:

```bash
curl -sL https://upsquadconnect.com/<route> | grep -oE "<expected-string>"
```

If the change isn't there, check `pm2 logs upsquad --lines 30 --nostream` or re-inspect `server/public/<route>/index.html` on the VPS.

## Architecture Reference

| Piece | Value |
|---|---|
| VPS | `root@72.61.245.97` (shared with squadhub) |
| Deploy dir | `/root/upsquad` (git clone of `origin/main`) |
| Process manager | pm2, process name `upsquad`, ID 0 |
| Entry script | `/root/upsquad/server/index.js` (Express, port 3000) |
| TLS + routing | squadhub's Caddy (in `/opt/squadhub`) → `host.docker.internal:3000` |
| Live URL | https://upsquadconnect.com (also `www.upsquadconnect.com`) |
| Admin URL | https://upsquadconnect.com/admin/login |
| Static build output | `/root/upsquad/server/public/` (gitignored; regenerated by `npm run build`) |
| Env file | `/root/upsquad/server/.env` (NOT committed) |

## Pitfalls

- **Main-branch hook** blocks edits on `main`. If you see it, `git checkout -b <name>` and retry.
- **Stale committed Caddyfile**: `/opt/squadhub/Caddyfile` as committed in the squadhub repo is out of date vs. what's actually running on the VPS. Don't trust it — read the live file on the VPS if routing questions come up.
- **Data-file regressions**: after any refactor that splits `Pricing.jsx` apart (e.g., into `client/src/data/pricing.js` + `client/src/components/pricing/*`), re-verify every field the renderer reads (`bold`, `sub`, `approach`, `weekly`) is still populated. The `weekly` field on the Capacity row has been dropped once already.
- **Two deploy scripts exist**: `tools/deploy.sh` is the *planned* Docker flow (not live). The current production is the pm2 flow above.
